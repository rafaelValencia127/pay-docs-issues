Para replicar el error se debe de mandar la petición con un token que respete la sintaxis del token pero que ya haya expirado.

Get /auth/generate-api-keys

Https status: 406 Not Acceptable

Para replicar este error se debe de superar la cantidad de api key permitida por usuario, para nuestra caso serían 5.

return actual:

{
    "message": "Key creation limit reached (5). Please delete at least one from your profile to create a new one."
}

Aplicando estandares:

Para este tipo de return, se considera que el mejor status https sería 409 conflict, este error nos indica que la peticíon no se pudo completar porque hay un confilto con el estado actual del recurso.

En este caso , se estaría violando una regla del negocio si se permite crear una llave más apartir de ya tener 5 keys, no sé recomienda en este caso el 406 ya que en general el estandar para este código es, cuando el cliente, mandá un body o hace una petición en un formato no valido para el servidor o viceversa (Esto lo sabemos por lo que colocamos en el header con el key, Accept)

Response Headers:

  • Status: 409 conflict
  • Content-Type: application/problem+json
  • Accept: application/json
{
  "type": "https://docs.adamopay.com/errors/key-limit-reached",
  "title": "Conflict",
  "status": 409,
  "detail": "Key creation limit reached (5). Please delete at least one existing key from your profile before attempting to create a new one.",
  "code": "API_KEY_LIMIT_EXCEEDED",
  "instance": "/auth/generate-api-keys"
}

Https status: 401 Unauthorized

Se esta presentando que se tiene varios estrucutras y mensajes para el mismo error, el caso del Unauthorized, por ejemplo para otros endpoint se tiene

Para el enpoint /auth/generate-api-keys

{
    "message": "Not Authorized: Expired. TokenExpiredError: jwt expired"
}

Lo ideal es tener el mismo JSON para los errores de la misma naturaleza

Error de consistencia

Ambas devuelven el mismo objeto, sin embargo hay un valor que no es consistente, currency dentro del objeto balanceDistributionInWallets, tenemos que su valor es en mayuscula o minuscula dependiendo del endpoint

/system-users/current-balance

{
    "currency": "cop",
    "balanceDistributionInWallets": [
        {
            "identifier": "8ebc28879059",
            ...
            "currency": "COP" //Valor en mayuscula
        }
    ],
    "totalBalance": 73596000,
    "balanceDescription": {
        ...
    }
}

/system-users/current-balance/{{currency}}

{
    "currency": "cop",
    "totalBalance": 73596000,
    "balanceDescription": {
        ...
    },
    "balanceDistributionInWallets": [
        {
            "identifier": "8ebc28879059",
            ...
            "currency": "cop" //Valor en minuscula
        }
    ]
}

Get payments/supported-document-types/{country}

El return de Chile devuelve en el body supportedBanks, cuando debería de ser documentos de personas naturales

{
  "country": "chi",
  "supportedBanks": [
    {
      "type": "RUT",
      "name": "RUT"
    }
  ]
}

Correcto

{
  "country": "bra",
  "supportedDocumentTypes": [
    {
      "type": "PA",
      "name": "Pasaporte"
    },
    {
      "type": "CPF",
      "name": "Cadastro de Pessoas Físicas"
    },
    {
      "type": "CNPJ",
      "name": "Cadastro Nacional da Pessoa Jurídica"
    }
  ]
}

Get /payments/supported-bank-account-types/{{country}}

El return tanto de chile , Peru y Brasil tenemos que en el objeto tiene un nombre de una llave así supportedBanks, pero Colombia, Mexico y Argentina tienen otro nombre la llave supportedBankAccountTypes , haciendo que el endpoint tenga inconsistencias

Brazil , Chile y Peru

{
    "country": "bra",
    "supportedBanks": [
        {
            "type": "checking",
            "name": "Checking account"
        },
        {
            ...
        }
    ]
}

Argentina, Colombia y Mexico

{
    "country": "col",
    "supportedBankAccountTypes": [
        {
            "type": "27",
            "name": "Checking account"
        },
        {
            ...
        }
    ]
}
